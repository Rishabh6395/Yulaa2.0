Data Flow

POST /login → auth.service.login() → users + user_roles + schools → bcrypt.compare(password, hash) → signToken({ id, email, roles }) → Return token + user object
OTP Reset: → Redis: otp:{userId} (6-digit, 10 min TTL) → email.service: sends HTML email via Brevo SMTP → verify-otp: Redis lookup + delete on success → reset-password: bcrypt hash + DB update

Security Notes

  • JWT stored in HTTP-only cookie (token) OR Authorization header (Bearer …)
  • Consultant login is blocked if contract status is expired or terminated
  • Error message is always generic ("Invalid email or password") — no user enumeration
  • Token default expiry: JWT_EXPIRY env var (default 7 days)

Data Flow — Application Submission

POST /apply → Validate school (admissionsOpen=true) → Phone uniqueness check (per school) → runValidation(): Aadhaar format, grade existence (GradeMaster), duplicate Aadhaar, duplicate name+DOB → Fetch active AdmissionWorkflow → Create AdmissionApplication + AdmissionChild rows → Return { applicationId, flags, riskScore }

Data Flow — Final Approval → Provisioning

POST /action { action: 'approved' } → processAction(): advance workflow step → Last step reached? → provisionApprovedApplication() → Create User + Parent + ParentStudent + Student → Mark application approved

Validation Logic

  • Aadhaar — must be exactly 12 digits (regulatory, always enforced)
  • Duplicate Aadhaar — checked against students table and non-rejected admission_children
  • Grade existence — applied grade must match school's GradeMaster. If GradeMaster is empty (school hasn't configured grades), this check is skipped
  • Duplicate name + DOB — warning only; admins decide
  • Errors (severity=error) score +30 risk; warnings score +10. Max score = 100
Grade-age check now uses GradeMaster from DB. If the school hasn't configured grades, the check is silently skipped (no more hardcoded age-range map).

Timetable & Scheduling Module

Files & Routes

RouteMethodDescription
/api/timetable/teacherGETTeacher's schedule for a date: assigned + substitute + reassigned-away slots
/api/timetable/reassignGET/POST/DELETEList / create / cancel TimetableReassignments
/api/timetable/logPOSTLog whether a class actually occurred (TimetableLog)
/api/scheduleGET/POSTSchool-level schedule slots for a class
/api/super-admin/schools/[id]/timetableGET/POST/DELETEAdmin: manage timetables, bulk upload, proxy reassignment

Data Model

Timetable (schoolId, classId, academicYear) → TimetableSlot (dayOfWeek, periodNo, startTime, endTime, subject, teacherId) → TimetableReassignment (slotId, originalTeacherId, substituteTeacherId, startDate, endDate, isActive) → TimetableLog (slotId, teacherId, date, status)

Teacher Daily View Logic

GET /api/timetable/teacher?date=2026-05-10 → Resolve teacher.id from user → assignedSlots = TimetableSlot WHERE teacherId=me AND dayOfWeek=Saturday AND timetable.academicYear=current → For each: check if active TimetableReassignment exists (startDate ≤ date ≤ endDate) → substituteSlots = TimetableReassignment WHERE substituteTeacherId=me AND isActive AND date in range → Return slotType: 'assigned' | 'reassigned_away' | 'substitute'

Reassignment Safety Guards

  • Substitute must belong to same school (teacher.schoolId check)
  • Substitute must not have approved leave during the date range (LeaveRequest check)
  • No overlapping active reassignment on same slot and date range
  • Substitute is notified on assignment AND on cancellation (notification.service)
  • DELETE: schoolId guard via slot.timetable.schoolId — prevents cross-school cancellation

Leave → Reassignment Cascade

Teacher withdraws approved leave → withdrawLeave() → Find TimetableReassignments WHERE originalTeacherId=teacher AND dates overlap AND isActive → Update isActive=false for all → Notify each substitute via notification.service

Cross-Module Links

  • Leave — substitute leave check at reassignment creation; leave withdrawal cascades to reassignment cancellation
  • Attendance — TimetableLog tracks if period was conducted (different from student attendance)
  • Students — student weekoff days derived from timetable (days with no slots)

Monthly Report Data Flow

Bulk Upload Security

When bulk-uploading student attendance, each student_id in the CSV is validated against students WHERE id IN(...) AND classId=? AND schoolId=?. Rows with student IDs from other classes or schools are silently dropped.

Academic Year Computation

Holiday and weekoff lookup uses the auto-computed academic year: m >= 4 ? "${y}-${y+1}" : "${y-1}-${y}" — so May 2026 looks up year 2026-2027 automatically.

Cross-Module Links

  • Leave — approved student leave auto-creates attendance records with remarks='__leave__'; withdrawal deletes them
  • Holidays — HolidayCalendar is joined on every monthly report
  • Timetable — TimetableLog is separate (period-level); Attendance is day-level for students

Leave Lifecycle

Submit → status=pending, currentStep=0
Reviewer approves step 0 → currentStep++ OR status=approved (last step)
Reviewer rejects → status=rejected
Applicant withdraws pending/approved → status=withdrawn

Approval Step Validation

PATCH { action: 'review', id, action: 'approved' } → findLeaveById(id, schoolId) — scoped to school → Check status === 'pending' → Load LeaveWorkflow steps → Verify step.approverRole === reviewer's roleCode (if set) → Verify step.approverUserId === reviewer's userId (if set) → advanceLeaveStep() → returns new status

Leave Types

Leave types come exclusively from LeaveTypeMaster table (school-configured). If the school has not configured any leave types, GET /api/leave/types returns { types: [], configured: false } and the UI should show a "configure leave types" prompt.

Hardcoded fallback leave types (Sick, Casual, Other) have been removed. The system now returns an empty array with configured: false when no types are set up.

Effective Days Calculation

countEffectiveDays(schoolId, start, end, weekoffs) → For each calendar day: skip if weekoff day OR in HolidayCalendar → Student leave: weekoffs from class timetable (days with no slots) → Employee leave: weekoffs from school leave-config (HolidayCalendar _weekoff_ entries)

Post-Approval Side Effects

  • Employee leave approved → TeacherLeaveBalance.usedDays incremented by effective days
  • Student leave approved → attendance records created with remarks='__leave__' for each effective date
  • Leave withdrawn (approved employee) → TimetableReassignments in the leave period are cancelled; substitutes notified
  • Leave withdrawn (approved student) → auto-created attendance records deleted

Cross-Module Links

  • Attendance — leave approval creates/deletes attendance records
  • Timetable — leave withdrawal cascades to reassignment cancellation
  • Holidays — effective day count excludes school holidays
  • school-utils — assertParentOwnsStudent guards parent leave submissions

Both holidays and weekoffs live in HolidayCalendar:

  • Weekoffs — epoch dates 1970-01-04…1970-01-10, scoped to academicYear='_weekoff_'

Academic Year Auto-Detection

All routes: year query param defaults to currentAcademicYearLabel() → May 2026 → looks up 2026-2027 → January 2027 → looks up 2026-2027 → No year hardcoding anywhere in routes
All || '2025-2026' fallbacks have been replaced with || currentAcademicYearLabel() across: /api/holidays, /api/schedule, /api/super-admin/…/leave-config, modules/classes/class.repo.ts.

Access Control

  • Viewing: all authenticated users
  • Adding/editing holidays: super_admin or principal only (school_admin blocked)
  • Weekoff configuration: same as holidays

Infrastructure & Services

lib/prisma.ts — Database

  • Neon PostgreSQL via PG adapter (Pool: max 3 connections, 30s timeout for cold-start)
  • Prefers DIRECT_URL over DATABASE_URL to bypass PgBouncer for migrations
  • SSL: rejectUnauthorized=false (dev; set to true in prod)

lib/redis.ts — Caching

  • ioredis client with automatic fallback to in-process Map cache
  • Lock prefix: lock: | SCAN-based cache invalidation (non-blocking)
  • Logs one warning per 60s when Redis is unavailable

services/cache.service.ts — Cache Pattern

  • withCache(key, ttl, fn) — simple cache-aside
  • withCacheLock(key, ttl, fn) — prevents cache stampede (thundering herd)
  • TTLs (env-configurable): dashboard=120s, parent dashboard=60s, lists=300s

services/email.service.ts — Email

  • Brevo (Sendinblue) SMTP API
  • App name: NEXT_PUBLIC_APP_NAME env var
  • App URL: NEXT_PUBLIC_APP_URL env var
  • Dev mode: logs to console when BREVO_API_KEY not set

services/pdf.service.ts — PDF Generation

  • Puppeteer (headless Chrome) — dynamic import to avoid cold-start overhead
  • A4 format, 15mm margins, printBackground=true
  • Sandboxed: --no-sandbox flag for Docker/serverless environments

utils/pagination.ts

  • Default limit: 20, max: 100
  • parsePagination(searchParams) → { page, limit, skip }

utils/errors.ts — Error Handling

  • Prisma error codes mapped to HTTP status (P2025→404, P2002→409, P1001→503, etc.)
  • All API routes use handleError(err) in catch block
  • Dev mode includes stack traces in responses
